01 - ARQUITECTURA
Patron de Diseno: Bridge
Basicamente separamos la parte visual de la logica del juego

Bridge Pattern

Es el patron principal del proyecto. La idea es que la interfaz (Svelte) y el motor del juego (Phaser) van por separado. Podemos cambiar menus sin tocar el juego, y al reves. Se comunican a traves de los stores.

Componentes (Svelte)

Cada pantalla es un componente independiente: tiene su HTML, su logica y sus estilos. Landing, Lobby, CharacterSelect... todos son componentes .svelte que se renderizan segun la ruta actual.

Observer con Stores

Usamos los runes de Svelte 5 ($state() y $derived()) para el estado global. Cuando algo cambia en el store, todos los componentes que lo usan se actualizan solos.

Mensajes WebSocket

Para el modo online, tenemos un sistema de mensajes por WebSocket. Cada mensaje que llega del servidor pasa por un handler que decide que hacer con el. Asi gestionamos el chat, las salas y el estado de la partida.

game/

Aqui esta todo el motor del juego: fisicas, personajes, escenas y el input del teclado. No hay nada de Svelte, es Phaser puro. Asi mantenemos la separacion del Bridge.

lib/

Lo que comparten ambas capas: los stores reactivos, el cliente HTTP para la API y toda la gestion del WebSocket para el multijugador.

ui/

Las 10 paginas de la aplicacion (Landing, Login, Lobby, Room, CharacterSelect, StageSelect, Game, Results, Settings...) y el HUD que se superpone al canvas.

public/assets/

Los sprites pixel-art de Blaze, Phantom y Titan, mas los fondos de las 4 arenas urbanas (Callejon, Almacen, Estacion, Azotea).

04 - USABILIDAD
Diseno Responsive
La app se adapta a distintos tamanos de pantalla

Tailwind CSS 4

Todo el estilo se hace con clases utilitarias de Tailwind. Usamos breakpoints sm: y md: en todas las paginas para que el tamano de textos, botones y gaps cambie segun la pantalla.

Phaser Scale Manager

El canvas del juego usa Scale.FIT con CENTER_BOTH, asi que se reescala solo y se centra en cualquier resolucion manteniendo el aspect ratio.

Layouts que cambian

Los botones del menu pasan de vertical a horizontal (flex-col sm:flex-row). En la Room, el chat y la lista de jugadores van lado a lado en escritorio. Los sprites de personajes se hacen mas pequenos en movil.

Viewport dinamico

Usamos 100dvh para que la altura se adapte a moviles (donde la barra de direcciones quita espacio). Ademas hay estilos especificos para cuando la altura es muy baja (landscape).

05 - FORMULARIOS
Validacion con JavaScript
Validamos los datos del usuario en el cliente con la reactividad de Svelte

Login y Register

✓

Campos obligatorios

El boton de enviar se deshabilita si falta el email o el password. Svelte lo controla reactivamente con disabled={!email || !password}.

✓

Errores visibles

Si algo falla, aparece un mensaje en rojo debajo del formulario. Se renderiza con un bloque condicional {#if error}.

✓

Control de envio

Una variable loading evita que el usuario pulse el boton dos veces seguidas y muestra "Logging in..." mientras se procesa.

Lobby - Crear o unirse a sala

✓

Nombre obligatorio

Si el nombre esta vacio o solo tiene espacios, trim() lo detecta y muestra "Enter your name first". Se guarda en localStorage.

✓

Codigo de sala

Se convierte automaticamente a mayusculas con .toUpperCase() para evitar errores al compartir el codigo entre jugadores.

✓

Limites de longitud

El nombre tiene maxlength="16" y el codigo de sala maxlength="8" directamente en el HTML para que no se puedan exceder.

06 - DATOS
Base de Datos
Modelos de datos y como se comunica el frontend con el backend

Como fluyen los datos

El frontend no guarda datos directamente, todo pasa por el backend:

  • API REST en localhost:8080/api para login y perfiles
  • WebSocket en localhost:9001 para el juego en tiempo real
  • LocalStorage solo para el nombre del jugador y ajustes
